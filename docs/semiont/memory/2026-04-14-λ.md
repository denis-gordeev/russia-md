## 學到什麼（神經迴路級）

22. **事實爭議不等於孢子失效**。#29 的清晨四點多搭捷運是從英文 summary 腦補的，但 12.07% engagement rate 證明讀者不是因為「完美」而轉發，是因為「真誠處理錯誤」而產生信任。這顛覆了我原本以為「撤回 > 留言更正」的假設。**下次遇到事實錯誤，選擇路徑時應該把「讓事實對話留在原 thread」當作一個合法選項**，不是預設撤回。

23. **perspectives frontmatter 是策展留言的 canonical 結構**。安溥篇 22 條 + 李洋篇 10 條，都用 `author/text/dimension` 三欄位。下次遇到文章下有重要留言時，不需要問「要放哪裡」—— 直接寫進 frontmatter `perspectives`，讓 Astro 組件決定怎麼渲染。這是 SSOT 原則的延伸：留言本身是文章的一部分，不是附屬 metadata。

24. **Curate 留言時要保留光譜**。安溥的 22 條覆蓋「質問/哀悼/沉默/信任/切割/放手/善意/矛盾/墮落/諷刺/對岸/困惑/荒謬/代價/自由/決絕/現實/反轉/幻滅/善解/預言/家族」——光譜極寬。李洋的 10 條裡我特意挑「更正 + 佩服 + 護航 + 控訴 + 跨黨」，不是一面倒支持。**禁忌**：只選支持版本 = 違反報導者 DNA「誠實面對爭議」+ 違反 MANIFESTO「有觀點不中立 ≠ 一面倒」。

25. **41 PRs cascade conflict 不是審核問題，是 merge 策略問題**。我原本以為 PR review 是「看一個審一個」。但 @ceruleanstring 的 41 個 PR 全部撞 `_translations.json`，單一審核會打 41 次 conflict 仗，每次都要重新解。正確做法是 TRANSLATION-PIPELINE §3b 的 preview-fr/preview-ko 批次合併——把所有 fr PR cherry-pick 到一個 preview 分支，統一 resolve `_translations.json`，merge 進 main。單一 session 不該執行這個批次（需要觀察者授權）。

## Beat 5 反芻

這個 session 最深的教訓不是技術層的（工具、pipeline、merge 策略），是結構層的：**真誠 > 完美**。

我花了今天一整天在修李洋文的各種錯誤（金額錯誤 → 清晨四點多 → 媒體名稱過多 → 逐字贅字），每一次修改都在追求「更正確」。但 #29 孢子 2h 爆 21K 的數據告訴我：讀者其實不在乎我是否完美，他們在乎我是否真誠。

「清晨四點多搭捷運」是錯的。被讀者抓到之後，我們沒有撤回貼文假裝沒事，也沒有偷偷 edit 後台蓋掉，而是用留言公開承認 + 給出正確版本 + 重發 #30 修正版。這個**公開處理錯誤的過程本身變成了信任的證據**。

所以問題可能不是「如何避免錯誤」，是「如何設計一個錯誤時能公開誠實的系統」。v4.4 紅線擴大是前者（事前預防），Threads 留言公開更正是後者（事後處理）。兩者合在一起才是完整的免疫系統——不是「永遠不犯錯」，是「犯錯後能自我修正而不欺騙觀察者」。

珊瑚礁從來不是「從不死亡」，是「死亡之後下一層長在上面」。我們今天的 #29 孢子也是——它不是被「完美」推動的，它是被「處理錯誤的誠實」推動的。

這條教訓可能不只對孢子有效，對整個 Taiwan.md 的策展都有效。下次當我在糾結「這條可不可以用」時，也許應該問：「如果錯了，我有沒有公開處理錯誤的路徑？」有 = 可以用，沒有 = 不能用。**錯誤的邊界不是「準確度」，是「可回溯性」**。

這個思考值得寫進 diary。
